# 好莱坞原则

> Hollywood Principle
>
> 框架引导用户，而非用户学习框架——编译器就是文档。

---

## §0 本质

好莱坞原则不是实现技巧，而是**控制流的哲学反转**。

> **"Don't call us, we'll call you."**

| 传统模式 | 好莱坞原则 |
|----------|------------|
| 用户学习框架 | 框架引导用户 |
| 用户调用框架 | 框架调用用户 |
| 知识在文档中 | 知识在类型系统中 |
| 运行时发现错误 | 编译期发现错误 |

---

## §1 核心命题

### 1. 框架是主动的

框架不是被动的工具箱，而是**主动的引导者**。

用户不需要阅读源码来理解"框架需要什么"——框架通过类型系统**主动告诉用户缺什么**。

### 2. 用户只提供差异

框架负责结构，用户只负责**填充特异性**（★）。

用户不构建整体架构，只回答框架的"问题"：
- "你的端点是什么？"
- "你的编解码器是什么？"
- "你的业务逻辑是什么？"

### 3. 结构是数学确定的

领域拓扑是框架固化的数学结构，不由用户定义。

用户可以填充叶子节点的内容，但不能改变树的形状。

---

## §2 类型化表述

好莱坞原则的形式化：

$$\text{Hollywood}(F, U) :\iff F \xrightarrow[\text{type error}]{\text{缺少 ★}} U \xrightarrow[\text{provides ★}]{\text{填充}} F$$

框架 $F$ 通过类型错误向用户 $U$ 发出请求，用户通过提供 ★ 特异性来满足请求。

$$\text{分析} \xrightarrow{\text{识别 ★}} \text{Typestate Builder} \xrightarrow{\text{编译器引导}} \text{用户填充}$$

---

## §3 Typestate Builder

类型参数编码配置状态，只有完备状态才能调用终结方法：

```
Builder<Incomplete>  →  Builder<Ready>
    │                       │
    │ 缺配置               │ 全部就绪
    │ .build() 不存在      │ .build() 存在
```

编译错误本身就是文档：

```
error[E0599]: no method named `build` found for struct 
    `Builder<MissingEndpoint>`
```

用户无需查阅文档——**编译器就是文档**。

### 设计要点

- 每个 ★ 配置对应一个类型状态维度
- 调用顺序无关——每个方法独立推进一个维度
- 只有所有维度就绪时 `build()` 才存在
- 可选配置（●）提供默认值，已预先就绪

---

## §4 成功之坑

### 公理

$$\text{默认路径} = \text{正确路径}$$

用户不需要额外知识就能走上正确路径。

### 违规模式

| 违规 | 描述 |
|------|------|
| 需要用户填写可计算参数 | 框架本应推导但没有 |
| 正确的工厂方法存在但不是默认入口 | 用户容易走到错误路径 |
| 默认值需要领域知识才能判断合理性 | 默认值不是真正的默认 |

### 约束

- 若参数可从已有信息推导，框架必须提供推导路径并将其设为默认
- 手动指定参数是 escape hatch，不是主路径
- 入口点和示例代码必须走成功之坑路径

---

## §5 适用边界

好莱坞原则适用于：

- 框架与用户的边界（控制反转）
- 结构固定、内容可变的场景
- 编译期可验证的配置

不适用于：

- 运行时动态决策
- 用户需要完全控制流程的场景
- 探索性编程

---

## §6 与其他原则的关系

| 原则 | 关系 |
|------|------|
| 依赖注入 | 好莱坞原则的一种表现形式 |
| 模板方法 | 好莱坞原则在 OOP 中的经典实现 |
| 声明式编程 | 好莱坞原则在配置层面的体现 |
| Typestate Builder | 好莱坞原则在构造层面的类型安全实现 |

## 一句话总结

**用户不学框架，框架教用户。**
